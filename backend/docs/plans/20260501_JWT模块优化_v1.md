## 1. 核心思路 (Mindset)

### 1.1 当前存在的问题分析

经过代码审查，当前JWT模块存在以下核心问题：

#### 问题一：安全密钥管理缺陷
- **现状**：`SigningKey` 直接从配置文件读取，无密钥轮换机制
- **风险**：一旦密钥泄露，所有已签发的Token都将失效，无法平滑过渡
- **影响**：生产环境密钥轮换需要强制所有Token失效，用户体验差

#### 问题二：Token泄露被动防御不足
- **现状**：虽然有长短Token+白名单机制，但缺少主动撤销能力
- **风险**：用户主动登出或检测到账号异常时，无法立即使已签发的Token失效
- **影响**：无法满足"单点登出"和"异地登录检测"等安全需求

#### 问题三：并发刷新场景处理不当
- **现状**：`RefreshToken` 生成的短Token没有加入Redis缓存，刷新过程中并发请求可能导致：
  1. 多个并发刷新请求产生多个不同的短Token
  2. 黑客可能在刷新窗口重放窃取的Token
- **影响**：Token有效性时间窗口被实际延长，增加被窃取后的可利用时间

#### 问题四：审计与可观测性缺失
- **现状**：签发、验证、刷新Token无任何日志记录
- **影响**：无法追踪Token生命周期，安全事件发生后难以回溯

### 1.2 优化方案逻辑闭环

#### 核心设计原则
1. **最小权限原则**：短Token仅用于请求认证，不包含刷新能力
2. **主动可撤销**：所有有效Token状态在Redis中可查询，支持主动撤销
3. **平滑轮换**：支持多个SigningKey同时有效，实现无痛密钥轮换
4. **并发安全**：刷新Token时生成新的短Token并同步到Redis，避免并发产生多个有效Token

#### 数据流动逻辑

```
登录流程:
 1. 用户提交认证信息 → AuthService.GetTokenPair()
 2. 生成 claims(短Token) 和 refreshClaims(长Token)
 3. 签发 shortToken 和 refreshToken
 4. Redis写入: 
    - whitelist:userID:refreshToken = "1" (TTL = RefreshExpiresTime)
    - tokens:userID:shortToken = JSON{role, issuedAt} (TTL = ExpiresTime)
 5. 返回: shortToken, refreshToken, expiresAt

请求认证流程:
 1. 请求携带 shortToken → JWTAuth middleware
 2. ParseToken 验证签名和时效
 3. Redis校验: GET tokens:userID:shortToken
 4. 验证通过 → 设置 context(userId, role)
 5. 验证失败 → 返回 401

刷新流程:
 1. 客户端提交 refreshToken → RefreshToken()
 2. 解析验证 refreshToken (检查IsRefresh=true)
 3. Redis校验: EXISTS whitelist:userID:refreshToken
 4. (关键) 生成新的 shortToken
 5. Redis写入新Token: tokens:userID:newShortToken (TTL = ExpiresTime)
 6. (可选) 删除旧的短Token缓存
 7. 返回: newShortToken, newExpiresAt

登出流程:
 1. 用户调用登出接口
 2. 从Redis删除: whitelist:userID:refreshToken
 3. 主动撤销: DEL tokens:userID:*
 4. 返回成功

密钥轮换流程:
 1. 新增 kid(Key ID)到配置
 2. 签发新Token使用新密钥
 3. 验证时按kid查找对应密钥
 4. 旧Token自然过期后移除旧密钥
```

## 2. 方案详述 (Solution Detail)

### 2.1 实现方式对比

#### 方案A：JWS + Key ID (jwt-go库)
- **优点**：标准协议，广泛支持，便于多服务间互操作性
- **缺点**：依赖外部库，增加维护成本
- **适用场景**：需要与其他系统集成的场景

#### 方案B：自定义Header + 密钥轮换
- **优点**：完全控制，实现简单，无外部依赖
- **缺点**：非标准协议，可能影响互操作性
- **适用场景**：封闭系统，当前项目现状

**选择方案B**，理由：
1. 当前项目已使用自定义JWT实现（`pkg/utils/jwt.go`）
2. 无外部系统集成需求，互操作性不是首要考虑
3. 实现成本低，与现有代码兼容性好

### 2.2 核心优化项

| 优化项 | 变更类型 | 优先级 |
|--------|---------|--------|
| 短Token加入Redis缓存，支持主动撤销 | Add | P0 |
| 密钥轮换机制 (kid头部) | Modify | P0 |
| 刷新Token并发安全控制 | Modify | P1 |
| 登出接口实现 | Add | P1 |
| Token生命周期日志 | Add | P2 |
| Redis Key命名规范统一 | Modify | P2 |

## 3. 详细执行步骤 (Execution Roadmap)

### Task 1: 扩展配置结构支持密钥轮换
**文件**: `config/jwt.go`

**变更内容**:
1. 添加 `CurrentKeyID` 字段 (当前生效的密钥ID)
2. 修改 `SigningKey` 为 `map[string]string` (支持多个密钥)
3. 添加 `KeyRotationInterval` 配置项 (密钥轮换间隔，天)

**效果**: 支持多密钥同时有效，实现平滑切换

### Task 2: 增强JWT Claims结构
**文件**: `pkg/utils/jwt.go`

**变更内容**:
1. 添加 `KeyID string` 字段到 CustomClaims
2. CreateToken时设置 kid 自定义header
3. ParseToken时根据kid查找对应密钥

**效果**: Token携带密钥标识，验证时自动选择正确版本

### Task 3: 实现Token缓存与撤销逻辑
**文件**: `pkg/utils/cache/jwt_cache.go` (新文件)

**变更类型**: Add

**核心接口**:
```go
type JWTCache interface {
    CacheAccessToken(ctx context.Context, userID uint, token string, info TokenInfo, ttl time.Duration) error
    ValidateAccessToken(ctx context.Context, userID uint, token string) (bool, *TokenInfo, error)
    CacheRefreshToken(ctx context.Context, userID uint, token string, ttl time.Duration) error
    ValidateRefreshToken(ctx context.Context, userID uint, token string) (bool, error)
    RevokeUserTokens(ctx context.Context, userID uint) error
    RemoveAccessToken(ctx context.Context, userID uint, token string) error
}
```

**效果**: 统一Token缓存管理，支持灵活的查询和撤销操作

### Task 4: 优化AuthService.GetTokenPair
**文件**: `service/auth_service.go`

**变更内容**:
1. 生成短Token后立即调用 JWTCache.CacheAccessToken()
2. 使用配置中的 CurrentKeyID 签发Token
3. 添加签发日志记录

**效果**: 短Token状态可查，支持主动撤销

### Task 5: 实现刷新Token并发控制
**文件**: `service/auth_service.go`

**变更内容**:
1. RefreshToken中添加 Redis 分布式锁 (`lock:refresh:userID`)
2. 获取锁后先尝试删除旧的短Token缓存
3. 生成新Token后调用 JWTCache.CacheAccessToken()
4. 释放锁

**效果**: 确保同一时间只有一个刷新请求成功，避免Token泛滥

### Task 6: 增强中间件验证逻辑
**文件**: `middleware/jwt.go`

**变更内容**:
1. ParseToken通过后，调用 JWTCache.ValidateAccessToken()
2. 验证失败返回 401 Unauthorized
3. 设置 context 时附加 role 信息

**效果**: 主动验证Token有效性，支持已撤销Token的拦截

### Task 7: 实现登出接口
**文件**: `service/auth_service.go` & `controller/auth_controller.go` (新文件/修改)

**变更内容**:
1. 新增 Logout 方法，调用 JWTCache.RevokeUserTokens()
2. 从请求中获取 userId (需先走中间件)
3. 删除所有相关的Token缓存

**效果**: 用户主动登出时立即使所有Token失效

### Task 8: 添加日志与监控
**文件**: `pkg/utils/jwt.go` & `service/auth_service.go`

**变更内容**:
1. CreateToken 记录签发事件 (userId, kid, tokenType)
2. ParseToken 记录验证失败事件 (类型: 拒绝原因)
3. 使用 logrus 结构化日志

**效果**: Token生命周期可追溯，安全事件可审计

### Task 9: 更新配置文档
**文件**: `settings.example.yaml`

**变更内容**:
1. JWT配置示例更新为多密钥结构
2. 添加 keyRotationInterval 配置项
3. 添加注释说明各字段用途

**效果**: 配置文档与代码实现同步

## 4. 核心代码要点 (Critical Snippets Only)

### CustomClaims 扩展
```go
// pkg/utils/jwt.go
type CustomClaims struct {
    UserID    uint   `json:"userId"`
    Role      string `json:"role"`
    IsRefresh bool   `json:"isRefresh"`
    KeyID     string `json:"kid"` // 新增：密钥版本标识

    jwt.RegisteredClaims
}
```

### 分布式锁刷新逻辑
```go
// service/auth_service.go - RefreshToken
lockKey := fmt.Sprintf("lock:refresh:%d", claims.UserID)
locked, err := global.GVA_REDIS.SetNX(ctx, lockKey, 1, 10*time.Second).Result()
if err != nil || !locked {
    return "", 0, errors.New("refresh in progress, please retry")
}
defer global.GVA_REDIS.Del(ctx, lockKey)

// 先移除旧的短Token缓存
_ = global.JWT_CACHE.RemoveAccessToken(ctx, claims.UserID, oldToken)

// 生成新Token
newClaims := j.CreateClaims(claims.UserID, claims.Role, false)
newToken, err := j.CreateToken(newClaims)

// 缓存新Token
global.JWT_CACHE.CacheAccessToken(ctx, claims.UserID, newToken, tokenInfo, ttl)
```

### 密钥轮换验证
```go
// pkg/utils/jwt.go - ParseToken
kid := token.Header["kid"].(string)
signingKey := getSigningKey(kid) // 根据kid查找对应密钥
if signingKey == "" {
    return nil, errors.New("invalid key id")
}
token.Claims = &CustomClaims{}
return token.SignedString([]byte(signingKey))
```
