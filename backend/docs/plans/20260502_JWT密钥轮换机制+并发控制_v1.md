## 1. 核心思路 (Mindset)

### 1.1 问题定义

当前JWT模块的**核心痛点**不是"无状态"与"有状态"的选择，而是以下两个实际问题：

#### 问题一：密钥泄露无法平稳过渡
- **现状**：使用单一SigningKey，所有Token共享同一个密钥版本
- **风险**：密钥泄露后无法在不影响所有用户的情况下更换密钥
- **影响半径**：密钥泄露 = 所有用户被迫登出，无法定向隔离

#### 问题二：并发刷新可能导致Token泛滥
- **现状**：RefreshToken方法缺少并发控制
- **风险场景**：
  ```
  1. 用户Token即将过期，多个API请求并发触发刷新
  2. 每个请求都生成了一个新的短Token
  3. 多个短Token同时有效，增加了被窃取后的攻击窗口
  ```
- **影响半径**：Token生命周期被意外延长，安全防护降级

### 1.2 优化原则

**核心决策**：保持短Token**纯无状态**（每次请求不查Redis），仅通过密钥轮换和并发控制增强安全性。

**实现边界**：
- ✅ 短Token：验证时只查签名，不查Redis（保持无状态）
- ✅ 长Token：刷新时查Redis白名单（保持现状）
- ✅ 密钥轮换：支持多版本密钥共存
- ✅ 并发控制：刷新Token时使用分布式锁
- ❌ 不实现短Token的主动撤销（接受自然过期）

## 2. 方案详述 (Solution Detail)

### 2.1 密钥轮换机制设计

#### Token结构与密钥版本标识
```go
type CustomClaims struct {
    UserID    uint   `json:"userId"`
    Role      string `json:"role"`
    IsRefresh bool   `json:"isRefresh"`
    jwt.RegisteredClaims
}

// Token Header 示例
{
    "kid": "20240501",    // Key ID，标识使用的密钥版本
    "alg": "HS256"
}
```

#### 密钥版本管理
- 配置中维护多个SigningKey，每个Key有一个唯一的ID（如日期字符串）
- 指定当前使用的KeyID（CurrentKeyID）用于签发新Token
- 验证时根据Token中的kid查找对应的SigningKey

#### 轮换策略
```
1. 引入新密钥：
   - 添加 { "20240501": "new-secret-key" }
   - 启用 CurrentKeyID = "20240501"
   - 旧密钥仍保留，用于验证旧Token

2. 等待自然过渡（例如30天）：
   - 新请求使用新密钥签发Token
   - 旧Token继续使用旧密钥验证
   - 随着时间推移，旧Token逐渐自然过期

3. 移除旧密钥：
   - 确认旧密钥签发的Token都过期后
   - 从配置中删除旧密钥
```

### 2.2 并发控制设计

#### 问题场景
```
时刻 T0: 短Token即将在5分钟后过期
时刻 T1: 客户端发起10个并发API请求
时刻 T2: 10个请求都检测到Token快过期，并发调用RefreshToken
结果：    生成了10个不同的短Token，都有效
风险：    Token的有效时间窗口被10倍延长
```

#### 解决方案：Redis分布式锁
```
1. 刷新前获取锁: lock:refresh:{userID} (TTL 10秒)
2. 获取成功：执行刷新逻辑
3. 获取失败：直接返回错误或等待重试
```

```go
func (j *JWT) CreateToken(claims CustomClaims, keyID string) (string, error) {
    token := jwt.NewWithClaims(jwt.SigningMethodHS256, claims)
    token.Header["kid"] = keyID // 添加密钥版本标识
    signKey := j.getSecret(keyID)
    return token.SignedString([]byte(signKey))
}

// 新增辅助方法
func (j *JWT) getSecret(keyID string) string {
    if key, exists := j.SigningKeys[keyID]; exists {
        return key
    }
    // 兜底：返回当前默认密钥
    return j.SigningKeys[j.CurrentKeyID]
}
```

**效果**: 生成的Token携带密钥版本标识，验证时可区分不同版本

### Task 3: 更新ParseToken验证逻辑
**文件**: `pkg/utils/jwt.go`

**当前逻辑缺陷**：
```go
// 当前只支持单一密钥验证
token, err := jwt.ParseWithClaims(tokenString, &claims, func(token *jwt.Token) (interface{}, error) {
    return []byte(j.SigningKey), nil // 只有一个密钥选择
})
```

**优化后逻辑**：
```go
func (j *JWT) ParseToken(tokenString string) (*CustomClaims, error) {
    var claims CustomClaims

    token, err := jwt.ParseWithClaims(tokenString, &claims, func(token *jwt.Token) (interface{}, error) {
        // 验证签名算法，防止算法混淆攻击
        if _, ok := token.Method.(*jwt.SigningMethodHMAC); !ok {
            return nil, fmt.Errorf("unexpected signing method: %v", token.Header["alg"])
        }

        // 从Token头部提取kid
        kid, ok := token.Header["kid"].(string)
        if !ok {
            return nil, errors.New("token missing key id")
        }

        // 根据kid查找对应密钥
        secret, exists := j.SigningKeys[kid]
        if !exists {
            return nil, fmt.Errorf("invalid key id: %s", kid)
        }

        return []byte(secret), nil
    })

    if err != nil {
        return nil, err
    }

    if claims, ok := token.Claims.(*CustomClaims); ok && token.Valid {
        return claims, nil
    }

    return nil, errors.New("invalid token")
}
```

**效果**: 验证时根据Token中的kid自动选择正确版本的密钥

### Task 6: 实现刷新Token并发控制
**文件**: `service/auth_service.go`

**新增逻辑（RefreshToken方法）**:
```go
func (s *AuthService) RefreshToken(ctx context.Context, refreshToken string) (string, int64, error) {
    j := utils.NewJWT(/* ... */)

    // 1. 解析验证refreshToken
    claims, err := j.ParseToken(refreshToken)
    if err != nil {
        return "", 0, err
    }

    // 2. 校验是否为刷新Token
    if !claims.IsRefresh {
        return "", 0, errors.New("invalid refresh token type")
    }

    // 3. 校验Redis白名单
    whitelistKey := fmt.Sprintf("whitelist:%d:%s", claims.UserID, refreshToken)
    exists, err := global.GVA_REDIS.Exists(ctx, whitelistKey).Result()
    if err != nil || exists == 0 {
        return "", 0, errors.New("refresh token invalid or expired")
    }

    // ========== 新增：并发控制 ==========
    lockKey := fmt.Sprintf("lock:refresh:%d", claims.UserID)
    locked, err := global.GVA_REDIS.SetNX(ctx, lockKey, 1, 10*time.Second).Result()
    if err != nil || !locked {
        return "", 0, errors.New("refresh in progress, please retry")
    }
    defer global.GVA_REDIS.Del(ctx, lockKey)

    // ========== 并发保护区间结束 ==========

    // 4. 生成新的短Token（使用当前KeyID）
    newClaims := j.CreateClaims(claims.UserID, claims.Role, false)
    newToken, err := j.CreateToken(newClaims, global.GVA_CONFIG.JWT.CurrentKeyID)
    if err != nil {
        return "", 0, err
    }

    return newToken, newClaims.ExpiresAt.Unix(), nil
}
```

**效果**: 确保同一时间只有一个刷新请求成功，防止Token泛滥

### Task 8: 配置兼容性适配
**文件**: `config/config.go` 或新建 `config/adapter.go`

**目的**: 支持从旧配置平滑迁移到新配置

```go
// 在Config初始化时添加适配逻辑
func adaptJWTConfig(jwt *JWT) {
    // 如果存在新的 signing-keys 且 current-key-id 存在，无需处理
    if len(jwt.SigningKeys) > 0 && jwt.CurrentKeyID != "" {
        return
    }

    // 向后兼容：如果只有旧 signing-key，转换为默认版本
    if jwt.SigningKey != "" {
        if jwt.SigningKeys == nil {
            jwt.SigningKeys = make(map[string]string)
        }
        jwt.SigningKeys["legacy"] = jwt.SigningKey
        if jwt.CurrentKeyID == "" {
            jwt.CurrentKeyID = "legacy"
        }
    }
}
```

**效果**: 现有配置文件无需修改即可运行

## 6. 优化效果总结

| 优化项 | 优化前 | 优化后 |
|-------|-------|-------|
| 密钥泄露影响 | 所有Token同时失效 | 仅新密钥签发的Token受影响，旧Token平滑过渡 |
| 并发刷新风险 | 产生多个有效短Token，延长攻击窗口 | 分布式锁确保唯一刷新结果 |
| 配置迁移 | 无兼容方案 | 自动适配旧配置，平滑升级 |
| 短Token性能 | 每次请求只验证签名（无状态） | **保持不变** |
| Redis查询 | 仅长Token刷新时查询 | **保持不变** |
