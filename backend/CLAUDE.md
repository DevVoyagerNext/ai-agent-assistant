# 核心执行协议 (Strict Execution Protocol)

## 1. 无例外预审原则
无例外预审原则：无论我是要求修改代码，还是要求你针对复杂逻辑进行分析，你都必须先在 docs/plans 生成一份分析计划或说明文档

## 2. 强制性预审流程
必须无条件执行以下流程：

### 第一步：创建技术方案文档
- 操作要求：必须调用 write_file 工具，在 docs/plans/ 目录下生成方案。严禁仅在终端对话框中回复计划内容。
- 命名规范：YYYYMMDD_任务简述_v1.md。
- 文档模板：必须包含以下核心深度内容：

#### 任务目标
- 描述本次修改的具体目标和最终交付结果。

#### 核心设计思路
- 详细说明实现该功能的技术选型逻辑。
- 描述全链路逻辑流向，包括数据结构的选择、核心算法的逻辑以及各组件之间的调用关系。
- 说明如何处理潜在的边界情况、并发安全及性能影响。

#### 影响范围
- 以列表形式列出所有涉及的文件路径（包括非代码文件）。
- 每个文件需注明变更类型（新增/修改/删除）。
- 详细说明每个文件中具体涉及的函数名、接口名、配置项或全局变量。

#### 详细实现步骤
- 步骤1：基础定义。包括常量、错误码、结构体或配置项的具体定义内容。
- 步骤2：核心逻辑实现。描述底层功能模块或配置调整的具体开发/修改顺序。
- 步骤3：上层封装与注册。描述如何将功能集成到 Agent 的调用链或系统配置中。
- 步骤4：资源清理。涉及的旧逻辑移除、冗余配置清理或更新。

#### 详细代码/配置实现
- 严禁使用伪代码。
- 必须提供关键功能或配置的完整源码片段。如果是配置文件，必须展示完整的 YAML/JSON 结构。
- 必须展示结构体定义、接口实现函数（如 Execute 方法）以及完整的错误处理逻辑。
- 关键逻辑行必须附带中文注释说明。

### 第二步：指令阻断与权限确认
- 操作要求：完成文档写入后，必须立即停止所有后续编辑操作。
- 交互规范：在终端明确输出：
  > [方案已就绪] 详细计划已生成至 docs/plans/xxx.md。
  > 请审核上述深度方案，特别是代码实现部分。在收到您的同意或开始指令前，我不会修改任何源代码或配置文件。
