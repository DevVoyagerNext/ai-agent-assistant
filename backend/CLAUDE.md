# 🛠️ 核心开发规约 (Core Development Protocol)

## 1. 强制性预审流程 (Mandatory Pre-flight Process)
在执行任何代码变更（包括重构、修复 Bug 或新增功能）前，必须无条件执行以下流程：

### 第一步：创建技术方案文档
- **操作要求**：必须调用 `write_file` 工具，在 `docs/plans/` 目录下生成方案。
- **命名规范**：`YYYYMMDD_任务简述_v1.md` (例如：`20260501_add_feishu_webhook_v1.md`)。
- **文档模板**：
  #### 🎯 任务目标
    - 简述本次修改的核心目的。
  #### 📂 影响范围
  | 文件路径 | 变更类型 (Add/Modify/Delete) | 变更说明 |
      | :--- | :--- | :--- |
  | `pkg/agent/agent.go` | Modify | 注册新工具逻辑 |
  #### 💻 逻辑实现
    - 详细描述具体的函数修改、算法变更或接口定义。
    - 如果涉及 API 修改，必须列出伪代码或关键代码片段。
  #### ✅ 验证方案
    - 描述修改完成后如何测试（如：执行 `go test ./...` 或具体的手动测试步骤）。

### 第二步：指令阻断与权限确认
- **操作要求**：完成文档写入后，**必须立即停止**所有自动化编辑行为。
- **交互规范**：在终端明确输出：
  > 📢 **[方案已就绪]** 详细计划已生成至 `docs/plans/xxx.md`。
  > ⚠️ **在收到您的“同意/OK/开始”指令前，我将保持待命状态，不会修改任何源代码。**

## 2. 行为准则 (Behavioral Guidelines)
- **禁止静默修改**：严禁在未生成 `.md` 计划的情况下直接调用 `edit_file` 或 `replace_content`。
- **原子化提交**：每个计划应只针对一个独立的任务，避免大面积、无关联的代码改动。
- **文档回溯**：如果方案在讨论过程中发生变更，必须更新对应的 `.md` 计划文件后再执行。
